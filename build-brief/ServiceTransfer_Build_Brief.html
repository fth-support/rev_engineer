ServiceTransfer — Build Brief

This single document is your directive to (re)build ServiceTransfer from scratch in any language. It captures the full system — design, business logic, data relationships, configuration and security mandate — reverse-engineered from the original Visual Basic 6 source. Pair it with the repository folder it ships in and you have everything required to start implementation immediately, on any stack (Go, C#/.NET, Java, Node/TypeScript, Rust, Python…).

What you are building ServiceTransfer — a resilient background data-synchronization agent for a retail POS network. It runs at every branch, continuously pushing completed sales transactions from a local database up to a central headquarters database, tokenizing sensitive card data on the way, and posting member loyalty points to a member database.

How to use this brief

  1. Read sections 1–11 to fully understand the existing behaviour (the "what" and "why").
  2. Treat sections 12–13 (non-functional + security) as hard requirements for the new build.
  3. Follow sections 14–17 to design the target system, sequence the work, and verify you are done.
  4. Use the repository (see the map below) to confirm any detail directly against the original source.
Golden rule This is not a line-by-line port. Preserve the business rules exactly; replace the architecture (polling, direct DB access, string-built SQL, plain-text secrets) with the modern equivalents in §14–15.

1 · System overview

ServiceTransfer is a headless background service (the original hides its window and runs as a single instance). On a fixed timer it performs one work cycle: verify it should run, read which tables to sync, pull unsent rows from the local DB, write them to the central DB (tokenizing card fields en route), then post accumulated member points.

500msTimer poll interval (legacy)
3Databases: Local · Central · Member
0/1/3FTStaSentOnOff sync flags
SOAPSafeNet tokenizer (external)

Core responsibilities

  • Sales synchronization — move pending sales (header, detail, receipt, card, points, plus deposit/hold/voucher variants) from Local → Central.
  • Tokenization — replace raw card/member numbers with tokens via SafeNet before any sensitive value leaves the branch.
  • Member points — aggregate loyalty points per member and post them (with history) to the Member DB.
  • Resilience — survive offline periods, duplicate keys, timeouts; never send a header before its child rows.

Upstream producer — POSFront (the input contract)

ServiceTransfer does not create sales; it is the consumer half of a two-program pipeline. The producer is POSFront, the cashier-facing VB6 POS application (Adasoft, "TAKASHIMAYA project", v6.2412.1). The two never call each other — they hand off through the Local DB via two status flags. Note that POSFront's Sub Main is what launches this agent (alongside ServiceOnOff / ServiceAutoReplicate / ServiceAutoClear); a modern rebuild should instead run the sync service as an independent process.

FlagOwnerMeaning & handoff
FTShdStaDocPOSFront'2' in-progress → '1' complete (paid + receipt printed). An AFTER UPDATE trigger TRG_Tmp2Sale<Tmn> → STP_PRCxTmp2Sale then promotes the bill from the per-terminal working table TPSHD<Tmn> into TPSTSal* with FTStaSentOnOff='0'.
FTStaSentOnOffServiceTransfer'0' pending → '1' synced to HQ (this agent owns this flag).
Input contract The sync service consumes rows where FTShdStaDoc='1' AND FTStaSentOnOff='0'. A row still at '2' is an open bill and must be ignored. Master data (products, prices, members, tender) flows the other way — Server → branch — via the separate replication service.

ModernizeKeep the single-instance + config-load + connect + log steps, but replace the Registry state with a proper config/secret store, and the 500ms timer with an event trigger (§14).

6 · Sync cycle — the 500ms loop

One cycle (the original disables the timer while running to prevent re-entrancy):

1
Pre-check
Read Registry SentSale/Online. If either ≠ '1' → skip to Cleanup.
2
Connect & validate
Verify/repair Central & Local connections (health-check SELECT FTTmnCode FROM TCNMTerminalMtn); check store-open (FTTmsOpnStore='Y'). Any failure → Cleanup.
3
Configuration
Load active tables from TSysSync (ordered); resolve terminal number; load sale date from TSysChgDateTime.
4
Sync loop
FOR each table → fetch Local rows where FTStaSentOnOff='0' AND FDShdTransDate ≤ saleDate → FOR each row → Insert (§7); on success set the Local row's flag to '1'.
5
Member points
Call the points routine (§9).
6
Cleanup
Persist run state; re-enable the timer for the next cycle.
FOR each table IN activeSyncTables:
    rows = SELECT * FROM local.[table]
           WHERE FTStaSentOnOff = '0' AND FDShdTransDate <= saleDate
           ORDER BY <table PK>
    FOR each row IN rows:
        ok, err = InsertToServer(row, table)         // §7
        IF err == CONNECTION_LOST: GOTO Cleanup       // abort cycle
        IF err == DUPLICATE_KEY:  ok = UpdateOnServer(row, table)  // only if flag=='3'
        IF ok: SET local.[table].FTStaSentOnOff = '1' WHERE <row PK>

7 · Insert logic (the most safety-critical routine)

FUNCTION InsertToServer(row, table) -> (ok, errorCode):
  // (a) HD-FIRST RULE — a header may only go up after ALL its children are sent
  IF table is a Header (e.g. TPSTSalHD / TPSTDepositHD):
     FOR child IN [DT, RC, CD, SalePoint of this header]:
        IF EXISTS local child WHERE same keys AND FTStaSentOnOff <> '1':
           RETURN (false, 555)        // children not ready → skip header this cycle

  // (b) DELETE-BEFORE-INSERT — clean a stale partial record on the server
  IF EXISTS server row WHERE same keys AND FTShdStaDoc <> '1':
     DELETE server row WHERE same keys

  // (c) BUILD THE ROW (new build: typed parameters, NOT string concatenation)
  FOR each field IN row:
     value = field.value
     IF field needs tokenizing (see §8 matrix):
        value = Tokenize(value)        // may RETURN (false, 557) on failure
     // legacy formats by prefix (FT quote/escape, FC/FN numeric, FD date) — you use params

  // (d) INSERT INTO server.[table] (...) VALUES (...)

  // (e) POST-CHECK (headers): verify the header now exists on the server.
  //     If not → reset this header + all its children back to '0' and RETURN (false, 556)
  RETURN (true, 0)
Preserve exactly The HD-first rule, delete-before-insert, and header post-check reset are core data-integrity guarantees. They ensure HQ never holds a header without its children. Keep these as DB transactions in the new build.

8 · Tokenization

Sensitive card/member numbers must be tokenized before leaving the branch. The legacy call is SP_PRCbToken() against the SafeNet SOAP service; format FIRST_SIX_TOKEN keeps the first 6 digits and randomizes the rest.

FUNCTION Tokenize(cardNo) -> (token, ok):
  IF NOT tokenEnabled OR len(cardNo) < 7:   RETURN (cardNo, true)   // guard: pass through
  IF len(cardNo) < 14 OR cardNo is empty:   RETURN (cardNo, true)   // only tokenize len >= 14
  IF online AND not skipDbSearch:
     existing = SELECT TOP 1 FTToken FROM TPSTTokenLst
                WHERE FTValue = cardNo AND FTValue <> FTToken
                ORDER BY FDDateIns DESC
     IF existing found: RETURN (existing, true)        // reuse — avoids a web-service call
  IF InitSoap() ok:
     token = soap.InsertToken(naeUser, naePwd, dbUser, dbPwd, db, cardNo, FORMAT=4, luhn=0)
     IF error OR token empty: RETURN (cardNo, false)   // → caller raises Error 557, skips row
     INSERT INTO TPSTTokenLst (date, who, table, field, keys..., FTValue, FTToken, status=1, srv)
     RETURN (token, true)
  RETURN (cardNo, false)

9 · Member points

FUNCTION UpdateMemberPoints():
  REQUIRE online AND connect(Member) AND connect(Local)
  pending = SELECT DISTINCT keys FROM TPSTSalePoint WHERE FTRemark='0' AND FTSpoMemID <> ''
  FOR each transaction IN pending:
     groups = SELECT SUM(FCSpoPoint) AS pts, FTSpoMemID
              FROM TPSTSalePoint WHERE same trans AND FTRemark='0'
              GROUP BY FTSpoMemID, FDDateUpd, FTWhoUpd
     FOR each g IN groups:
        memberToken = Tokenize(g.FTSpoMemID)            // §8; also write token map to Member DB
        pts = g.pts
        IF transType IN {Return, IPV, DepositCancel, CancelVoucher, ...}: pts = -pts   // sign
        member = SELECT TOP 1 FROM TCNMMallCard WHERE FTMcdCode = memberToken
        IF member exists:
           UPDATE TCNMMallCard SET FCEarned += pts,
                  FCBalance = FCMcdTotalPoint + FCEarned + pts
        ELSE:
           INSERT TCNMMallCard (FTMcdStaAct='A', FTMcdName='New Member', FCEarned=pts, FCBalance=pts, expDate)
        INSERT INTO TPSTBPHis (...history row...)
        UPDATE TPSTSalePoint SET FTRemark='1' WHERE same keys      // mark processed
  ON ERROR: re-check web service + connections, then continue with next record

14 · Target architecture (the "to-be")

Shift from "agent opens databases across the network and polls" to "agent emits events; HQ services own the data".

Branch agent

Detect new/changed rows via change-data-capture or log-tailing instead of polling. Buffer locally (store-and-forward) and publish over TLS. Stays simple, stateless beyond its outbox.

Transport

A durable, replayable message broker (Kafka/RabbitMQ) decouples branches from HQ so a brief HQ outage loses nothing. At-least-once delivery + idempotent consumers.

HQ services

One API/consumer owns all writes inside DB transactions, with parameterized SQL/ORM, validation, and the same business rules (HD-first, dedupe, points). Central auth & audit.

Data & security

Tokenization via HSM/Cloud KMS (replace the legacy SOAP service); secrets in a vault; encryption in transit and at rest.

16 · Milestones

  1. Schema & contracts — model all tables (§4) + the event/payload schema. Stand up Local/Central/Member DBs (or test doubles).
  2. Config & secrets — boot from env/vault; connect to all three DBs; health checks (§11).

  7. Transport — move HQ writes behind the API/queue; add retries/backoff/dead-letter.
  8. Hardening — TLS, secrets rotation, metrics/alerts, load & failure testing; cut over branch-by-branch.

17 · Acceptance criteria (definition of done)

  • A sale recorded at a branch appears complete (header + all children) at HQ, or not at all — never a partial header.
  • No raw card/member number is ever stored or transmitted centrally; tokenization failure skips the row (never falls back to raw).
  • Re-running a cycle on already-synced data is a no-op (idempotent) — duplicates are handled, not duplicated.
  • Member point totals reconcile exactly with the source TPSTSalePoint rows (sign rules honoured), with a history entry per change.
  • A branch can be offline for hours and fully catch up on reconnect, in order, with nothing lost.
  • No string-built SQL, no plain-text secrets, no un-encrypted cross-WAN DB traffic anywhere.
  • All five legacy error conditions (555/556/557/dup/conn-lost) have explicit, tested handling.
