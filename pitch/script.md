# 🎤 Script — Evidence-Driven Modernization

**ใช้คู่กับ:** `Evidence_Driven_Modernization.html` (เปิดเต็มจอ 1 หน้า)
**เวลา:** ~4–5 นาที · **โทน:** มั่นใจ ตรงประเด็น มี proof
**คนฟัง:** หัวหน้า / ผู้บริหาร (เทียบกับทีมที่เสนอ Fujitsu "AI-Driven Modernization")

> เป้าหมายของสคริปต์นี้: ไม่ได้จะล้มของเขา แต่จะ **วางจุดยืนของเราให้คม** ว่าเราต่างกันตรงไหน และทำไม "เอกสารที่ถูกต้องตั้งแต่นับ 1" คือสิ่งที่โครงการ modernize ทุกตัวขาดไม่ได้

## 0) เปิด (15 วิ) — ตรึงด้วยประโยคเดียว
> "ทุกโปรเจกต์ modernize ที่ล้ม ไม่ได้ล้มตอนเขียนโค้ด — มันล้มตั้งแต่ **เข้าใจของเดิมผิด** ครับ
> สิ่งที่ผมจะเสนอวันนี้คือวิธีทำให้ 'นับ 1' ของเรา **ถูกต้อง 100%** ก่อนจะไปนับ 2"

*(คลิกเปิด one-page ค้างไว้)*

---

## 1) Framing — ยอมรับของเขา แล้วชี้ช่องว่าง (30 วิ)
> "ทีมข้าง ๆ เสนอ Fujitsu AI-Driven Modernization — สวยครับ ครบ lifecycle มี AI agent หลายตัว มี platform
> แต่ขอตั้งข้อสังเกต 3 อย่างที่เป็นความเสี่ยงจริงของแนวนั้น:
> 1. **AI-Driven** = ขับเคลื่อนด้วย AI ล้วน — ถ้า AI เข้าใจ legacy ผิด มันจะผลิตเอกสารผิด ๆ ออกมาอย่างมั่นใจ (hallucinate)
> 2. **Platform lock-in** — ผูกกับเครื่องมือ/LLM ของ vendor (Kozuchi, TAKANE) ออกยาก ค่าใช้จ่ายต่อเนื่อง
> 3. สังเกตมุมสไลด์เขาเขียนว่า **'For show on screen only'** — คือยังเป็นภาพ vision ไม่ใช่ของที่ proven"

---

## 2) จุดยืนของเรา — ชื่อ + ประโยคเด็ด (20 วิ)
*(ชี้หัวสไลด์)*
> "ของเราเรียกว่า **Evidence-Driven Modernization** ครับ
> เราใช้ AI เหมือนกัน — แต่ **ขับเคลื่อนด้วยหลักฐาน ไม่ใช่การเดา**
> สโลแกนคือ: **We don't guess — we verify.** ทุกแผนภาพ ทุก spec ทุกการตัดสินใจ ผม trace กลับไปที่ **โค้ดจริง** ได้หมด"

---

## 3) อ่าน Matrix — แกนนอน 6 สเตป (75 วิ)
*(สไลด์จัดเป็น **matrix แบบเดียวกับเขา**: คอลัมน์ = 6 สเตป, แถว = layer — ให้นายวางเทียบกันได้ตรง ๆ)*

> "อ่านจากบนลงล่างได้เลยครับ — แถวบนคือ 6 สเตป map ตรงกับ lifecycle ของเขา (Plan/Asset/Design/Implement/Test/Operate)"

1. **Discover** — อ่าน source เดิม "ตามที่มันเป็น" (ไม่ใช่เดาจากชื่อไฟล์) — เราอ่าน `wMain.frm` + `mMain.bas` จริง
2. **Verify** — นี่คือ **กระดูกสันหลัง (verification spine)** ที่เขาไม่มี — cross-check ทุก fact กับโค้ด
   > *"ตอนทำจริง ผมเจอ spec เดิมเขียน field ผิด และเขียนเงื่อนไข security กลับด้าน — ถ้าเชื่อ AI ดื้อ ๆ โครงการพังตั้งแต่ยังไม่เริ่ม"*
3. **Visualize** — ทำ **documentation portal แบบ interactive** คลิกดู architecture / ER / flow ได้จริง ไม่ใช่ภาพนิ่ง
4. **Specify** — สรุปออกมาเป็น **Build Brief ไฟล์เดียว ที่ภาษาอะไรก็เอาไปสร้างต่อได้ทันที** (pseudocode + business rules)
5. **Modernize** — ปรัชญาชัด: **เก็บ business rule เดิมเป๊ะ / เปลี่ยนเฉพาะ architecture** (polling→event, ต่อ DB ตรง→API+TLS)
6. **Handoff** — มี **acceptance criteria** บอกว่า "เสร็จ" หน้าตาเป็นยังไง วัดได้

## 3.1) อ่าน Matrix — แกนตั้ง (Layers) (30 วิ)
*(ชี้ไล่ลงทีละแถวซ้ายมือ — ตรงกับ layer ของเขาเป๊ะ: Orchestration / Agent / Features / Tools / **AI Model** / Knowledge / HITL)*
- **Orchestration** — agent ตัวเดียว (Claude) คุมครบทั้ง 6 สเตป → ไม่มีรอยต่อระหว่าง agent หลายตัวให้พัง
- **AI Agent** — แต่ละสเตปมี role เฉพาะ (Code Archaeologist → Verification → Docs → Analyst → Architect → QA)
- **Features** — ดู **จุดสีเขียว = Proven (ทำจริงกับ ServiceTransfer แล้ว)** ของเราเขียวเกือบหมด ของเขายังเป็น PoC/vision เยอะ
- **AI Model / LLM** — *(จุดที่นายถามพอดี)* ทุกสเตปใช้ **Claude Opus 4.x** เป็นหลัก และที่สเตป Build **เสียบ Copilot / GPT เพิ่มได้** — สื่อว่าเรา **เปิด ไม่ lock** ต่างจากเขาที่ผูก Kozuchi/TAKANE
- **Knowledge** — ใช้ **source repo จริงเป็น ground truth** ไม่ต้องมี external RAG ที่ drift ออกจากของจริง
- **Human-in-loop** — คน approve/ชี้ทิศ, AI ทำงานหนัก — review แบบ targeted ไม่เป็นคอขวด

## 4) Why this wins — 4 หมัดเด็ด (45 วิ)
*(ชี้แถบล่าง — win chips 4 อัน)*
> "ต่างจากเขา 4 ข้อหลัก:
> - **EVIDENCE** — grounded ไม่ใช่ generated · พิสูจน์แล้วว่าจับ error ของเดิมได้
> - **PORTABLE** — output ไม่ผูกภาษา/แพลตฟอร์ม · จะสร้างด้วย Go, .NET, Java, Node ก็ได้วันแรก
> - **INTERACTIVE** — เอกสารระดับใช้งานจริง สองภาษา (TH/EN) ไม่ใช่ 'for show only'
> - **LEAN** — เครื่องมือ open + AI ตัวเดียว · ไม่ต้องซื้อ platform หนัก ๆ มาเลี้ยง"

---

## 5) Proof — ปิดด้วยของจริง (30 วิ)
*(ชี้แถบล่างขวา — proof KPIs + บรรทัด ★ ServiceTransfer)*
> "และนี่ไม่ใช่ทฤษฎีครับ — เราทำจริงกับระบบ **ServiceTransfer** (POS sync agent ภาษา VB6 เก่า):
> - อ่าน + verify โค้ด **2,860 บรรทัด**
> - fact **100%** trace กลับ source ได้
> - ส่งมอบ **15+ เอกสาร**: เว็บ interactive, DOCX TH/EN, PPTX native
> - และสรุปเหลือ **Build Brief ไฟล์เดียว** ที่ทีมไหนรับไปก็เขียนใหม่ได้ทันที"

---

## 6) ปิดการขาย — "นับ 1" (20 วิ)
> "ข้อความเดียวที่อยากให้พี่จำกลับไปคือ —
> **เอกสารคือ step 1 ของการพัฒนาทุกอย่าง ถ้านับ 1 ผิด ทุกอย่างหลังจากนั้นผิดหมด**
> ทีมเขาเก่งเรื่อง 'สร้าง' ครับ แต่ของเราทำให้ '**ฐานราก**' ไม่มีวันพัง
> ผมขอเสนอให้ลองใช้แนวนี้กับ 1 ระบบจริงเป็น pilot — แล้วเทียบผลกันเลยครับ"

## 📌 Q&A — เตรียมรับคำถาม
| คำถามที่อาจโดน | คำตอบสั้น |
|---|---|
| "ของเขามี AI agent หลายตัว ของเราตัวเดียวพอเหรอ?" | "agent เยอะ = จุดพังเยอะ + ประสานยาก ของเราตัวเดียวที่ orchestrate ครบ ลด coordination overhead และ verify ได้ทั้งสาย" |
| "Scale กับระบบใหญ่ ๆ ได้ไหม?" | "Pipeline เดียวกัน รันซ้ำต่อ module/ต่อ service ได้ และ verification spine ยิ่งสำคัญเมื่อระบบใหญ่" |
| "ใช้ Claude ก็ผูก vendor เหมือนกันนี่?" | "ผูกแค่ขั้น 'ผลิต' ครับ แต่ **output เป็นมาตรฐานเปิด** (md/html/docx) ไม่ lock — สลับ LLM ได้ ของเขา lock ทั้ง platform" |
| "เร็วกว่าจริงไหม?" | "ServiceTransfer ทั้งระบบ จาก legacy → เอกสารครบ → build brief จบในรอบงานเดียว ไม่ต้องตั้ง platform" |
| "แล้วเรื่องคนล่ะ (HITL)?" | "คนยัง in control — approve/ชี้ทิศ ส่วนงานหนักให้ AI ทำ เหมือนเขา แต่เราเพิ่ม verification ให้คนเชื่อผลได้" |

## 🗒️ Tips การนำเสนอ
- เปิด **one-page เต็มจอ** ค้างไว้ตลอด — มันออกแบบให้เล่าจบในจอเดียว (ไม่ต้องสไลด์เยอะ = ตรงข้ามกับเขา)
- ถ้ามีเวลา เปิด **documentation portal จริง** หรือ **Build Brief.html** โชว์ 30 วิ ให้เห็นว่า "ของจริงจับต้องได้"
- เน้นเสียงหนักที่ 3 คำ: **Evidence · Verify · นับ 1**
